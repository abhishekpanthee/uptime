\chapter{EXPECTED OUTCOMES}

\section{Introduction}

This chapter outlines the anticipated results and deliverables upon successful completion of the Uptime Tracker project. The expected outcomes are derived from the objectives defined in Chapter 1 and the system design presented in Chapter 4.

\section{System Deliverables}

Upon completion, the project will deliver the following tangible outputs:

\subsection{Web Application}

A fully functional web application accessible via modern web browsers with the following capabilities:

\begin{enumerate}
    \item \textbf{User Management Module}
    \begin{itemize}
        \item Secure user registration with email verification
        \item Login/logout functionality with session management
        \item Password reset capability
        \item User profile management
    \end{itemize}
    
    \item \textbf{Website Monitoring Module}
    \begin{itemize}
        \item Interface to add, edit, and remove websites from monitoring
        \item Real-time status display showing current availability
        \item Response time visualization through interactive charts
        \item Historical uptime statistics and trends
    \end{itemize}
    
    \item \textbf{Public Status Page}
    \begin{itemize}
        \item Publicly accessible page showing status of opted-in websites
        \item Clean, minimal interface requiring no authentication
        \item Auto-refresh functionality for live updates
    \end{itemize}
\end{enumerate}

\subsection{Backend Services}

The project will deliver a robust backend system comprising:

\begin{enumerate}
    \item \textbf{RESTful API} providing endpoints for all application functionality
    \item \textbf{Monitoring Service} executing automated availability checks every 5 minutes
    \item \textbf{Data Aggregation Service} computing hourly and daily statistics
    \item \textbf{Authentication Service} managing JWT-based user sessions
\end{enumerate}

\subsection{Database System}

A PostgreSQL database storing:

\begin{itemize}
    \item User account information
    \item Website registration and ownership records
    \item Raw ping result data (response time, status, timestamp)
    \item Computed hourly and daily aggregates
    \item System configuration parameters
\end{itemize}

\subsection{User Experience}

The application interface is expected to provide:

\begin{itemize}
    \item Intuitive navigation requiring minimal learning curve
    \item Responsive design functioning on desktop, tablet, and mobile devices
    \item Clear visual indicators for website status (up/down)
    \item Informative error messages guiding user actions
    \item Fast page load times (< 2 seconds on standard connections)
\end{itemize}

\subsection{System Scalability}

The designed architecture should support:

\begin{itemize}
    \item Up to 100 concurrent users without performance degradation
    \item Monitoring of up to 500 websites from a single server instance
    \item Storage of 6 months of historical data per website
    \item Horizontal scaling capability for future growth
\end{itemize}

\section{Potential Impact}

Upon successful implementation, the Uptime Tracker system is expected to provide:

\begin{enumerate}
    \item \textbf{For Thapathali Campus:} A functional tool to monitor institutional web services, improving awareness of service availability and enabling proactive maintenance.
    
    \item \textbf{For Students and Faculty:} A transparent status page reducing uncertainty about service availability and providing a reference for checking institutional website status.
    
    \item \textbf{For System Administrators:} Historical data and analytics to identify patterns, optimize maintenance schedules, and demonstrate service reliability metrics.
    
    \item \textbf{As a Reference Implementation:} A documented open-source project that can serve as a learning resource for future DBMS and web development projects.
\end{enumerate}
