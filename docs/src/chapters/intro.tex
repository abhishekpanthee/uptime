\chapter{Introduction}

\section{Background}

Website availability is critical for organizations, with downtime costing businesses an average of \$5,600 per minute. Educational institutions like Thapathali Campus depend on web-based systems for student information, examination portals, and learning platforms. When these services experience downtime, users lack visibility into service status.

Commercial monitoring solutions require recurring costs, while open-source alternatives need technical expertise. The Uptime Tracker project aims to create an accessible web application for automated website monitoring using \gls{http}/\gls{https} protocols, historical analysis with \gls{sql} database, and public status pages built with modern \gls{api} architecture.

\section{Problem Statement}

Many institutions lack centralized systems to monitor web service status. Key challenges include:
\begin{itemize}
    \item \textbf{Lack of proactive monitoring:} Downtime goes undetected until users report issues
    \item \textbf{No historical visibility:} Administrators lack data on uptime patterns and trends
    \item \textbf{User uncertainty:} Users cannot determine if issues are service-related or local
    \item \textbf{Decentralized information:} No single authoritative status source exists
\end{itemize}

\section{Objectives}

\subsection{Primary Objectives}
\begin{enumerate}
    \item Design a web application for automated uptime monitoring using \gls{rest} \gls{api}
    \item Develop backend service performing \gls{http}/\gls{https} checks at 5-minute intervals
    \item Create \gls{dbms} schema using PostgreSQL for analytics data
    \item Build interactive dashboard with real-time status and \gls{ui} visualizations
    \item Implement user authentication with \gls{jwt} tokens
\end{enumerate}

\subsection{Secondary Objectives}
\begin{enumerate}
    \item Provide public status pages for registered websites
    \item Enable privacy controls for monitoring data visibility
    \item Generate aggregate statistics and trend analysis
\end{enumerate}

\section{Scope and Limitations}

\subsection{Scope}

The system includes: user management with authentication; website registration interface; automated HTTP/HTTPS monitoring every 5 minutes; PostgreSQL database for ping results and aggregates; analytics dashboard with visualizations; and public status pages with privacy controls.

\subsection{Limitations}

\begin{itemize}
    \item Limited to HTTP/HTTPS protocols only
    \item 5-minute check intervals (balance of speed vs. resources)
    \item Single-location monitoring
    \item No email/SMS alerting in initial version
\end{itemize}
