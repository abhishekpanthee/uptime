\chapter{SYSTEM ARCHITECTURE AND METHODOLOGY}

\section{Introduction}

This chapter presents the architectural design of the Uptime Tracker system, detailing the component structure, data flow, database schema, and development methodology. The architecture follows a three-tier pattern separating presentation, business logic, and data storage concerns, enabling independent development and maintenance of each layer.

\section{System Architecture Overview}

The Uptime Tracker system employs a modern web application architecture consisting of four primary components:

\begin{enumerate}
    \item \textbf{Frontend Application:} A Next.js-based web interface providing user interaction through dashboards, status pages, and management forms.
    \item \textbf{Backend API:} An Elysia.js server handling business logic, authentication, and data operations through RESTful endpoints.
    \item \textbf{Database:} PostgreSQL database storing user accounts, website registrations, and monitoring data.
    \item \textbf{Monitoring Service:} A background process executing scheduled availability checks and recording results.
\end{enumerate}

\subsection{Runtime Environment: Bun}

Bun serves as the JavaScript/TypeScript runtime for both frontend and backend applications. Key characteristics include:

\begin{itemize}
    \item Native TypeScript support without separate compilation step
    \item Significantly faster package installation compared to npm
    \item Built-in bundler and test runner
    \item Drop-in compatibility with most Node.js packages
    \item Optimized performance for web server workloads
\end{itemize}

\subsection{Frontend: Next.js}

Next.js provides the foundation for the user interface with the following features utilized:

\begin{itemize}
    \item \textbf{App Router:} File-based routing with layouts and nested routes
    \item \textbf{Server Components:} Reduced client-side JavaScript through server rendering
    \item \textbf{API Routes:} Backend-for-frontend patterns when needed
    \item \textbf{Built-in Optimization:} Automatic code splitting and image optimization
\end{itemize}

\subsection{Backend: Elysia.js}

Elysia.js handles API requests with these capabilities:

\begin{itemize}
    \item \textbf{Type Safety:} End-to-end TypeScript inference from route definitions
    \item \textbf{Validation:} Built-in request validation using TypeBox schemas
    \item \textbf{Middleware:} Plugin system for authentication, logging, and CORS
    \item \textbf{Performance:} Optimized for Bun runtime with minimal overhead
\end{itemize}

\subsection{Database: PostgreSQL}

PostgreSQL provides reliable data persistence with:

\begin{itemize}
    \item ACID compliance for data integrity
    \item Efficient temporal queries for time-series analysis
    \item Window functions for aggregation calculations
    \item JSON support for flexible data storage when needed
    \item Mature ecosystem with extensive tooling
\end{itemize}

\section{Monitoring Service Design}

\subsection{Check Execution Flow}

The monitoring service will operate as a background process executing the following workflow:

\begin{enumerate}
    \item \textbf{Schedule Trigger:} A scheduler triggers checks every 5 minutes (configurable via cron or interval timer).
    \item \textbf{URL Retrieval:} The service queries all registered URLs from the ownership table.
    \item \textbf{Parallel Execution:} HTTP requests are sent to all URLs concurrently with connection and response timeouts.
    \item \textbf{Result Recording:} For each URL, the service records response time (or null for failures) and status in the analytics table.
    \item \textbf{Aggregation Update:} After recording raw results, the service updates hourly aggregates as needed.
\end{enumerate}

\section{Development Methodology}

\subsection{Agile Approach}

The project follows an iterative development methodology with these characteristics:

\begin{itemize}
    \item \textbf{Incremental Development:} Features are implemented in small, functional increments
    \item \textbf{Regular Integration:} Code is frequently committed and integrated to identify issues early
    \item \textbf{Prioritized Backlog:} Features are prioritized by core functionality first, enhancements later
    \item \textbf{Testing Integration:} Each feature includes corresponding tests before moving to next
\end{itemize}

\subsection{Development Phases}

\begin{enumerate}
    \item \textbf{Phase 1 - Foundation:} Project setup, database schema, basic authentication
    \item \textbf{Phase 2 - Core Features:} Website registration, monitoring service, data recording
    \item \textbf{Phase 3 - Interface:} Dashboard implementation, status displays, visualizations
    \item \textbf{Phase 4 - Polish:} Public status pages, aggregation, error handling, testing
    \item \textbf{Phase 5 - Documentation:} Code documentation, user guide, deployment instructions
\end{enumerate}

\subsection{Version Control}

Git will be used for version control with the following practices:
\begin{itemize}
    \item Feature branches for isolated development
    \item Descriptive commit messages documenting changes
    \item Regular commits maintaining manageable change sets
    \item Main branch always in deployable state
\end{itemize}

\section{Deployment Architecture}

The system is designed for deployment on standard infrastructure:

\begin{itemize}
    \item \textbf{Single Server:} All components can run on one server for simple deployments
    \item \textbf{Separate Services:} Frontend and backend can be deployed separately for scaling
    \item \textbf{Database Hosting:} PostgreSQL can be self-hosted or use managed services
    \item \textbf{Process Management:} Background monitoring service managed via process managers
\end{itemize}

Minimum production requirements include a Linux server with Bun installed, PostgreSQL database access, and domain configuration for HTTPS termination.
